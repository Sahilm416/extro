---
title: Public assets
description: Ship static files and reference them from any surface.
---

Drop static files into a top-level `public/` directory and Extro ships them to the build output with their original names. Use it for images, fonts, JSON, or anything you reference by URL rather than import.

```
my-extension/
├── public/
│   ├── logo.svg
│   ├── img/
│   │   └── banner.png
│   └── data/
│       └── config.json
└── src/
    └── app/
        └── ...
```

`public/logo.svg` ships as `logo.svg` at the output root, `public/img/banner.png` as `img/banner.png`. Names and structure are preserved, never hashed.

## Referencing a public asset

The portable way is `chrome.runtime.getURL`, which resolves against the extension on every surface, including content scripts and the background service worker:

```tsx
<img src={chrome.runtime.getURL("logo.svg")} />
```

```ts
const config = await fetch(chrome.runtime.getURL("data/config.json")).then((r) => r.json())
```

On a routable surface (`popup`, `options`, `sidepanel`) you can also use a root-relative path, since the page loads from the extension origin:

```tsx
<img src="/logo.svg" />
```

A root-relative path does not work in a content script. The content script runs in the host page, so `/logo.svg` resolves against that site, not your extension. Use `chrome.runtime.getURL` there.

## Content scripts and `web_accessible_resources`

A content script can only fetch extension files that are declared web-accessible. When your project has a `content/` surface, Extro adds every public asset to `web_accessible_resources`, scoped to the same host matches as the content script. No configuration needed: drop a file, `getURL` it from your content script, done.

To narrow or override what is exposed, set `web_accessible_resources` yourself through `manifest` in `extro.config.ts`; it takes precedence.

## Not for imported assets

`public/` is for files referenced by stable URL. If you want an asset hashed and bundled with your code, import it from a source file instead and let Vite handle it:

```tsx
import logoUrl from "./logo.svg"

<img src={logoUrl} />
```

That path is unchanged and orthogonal to `public/`.

## Not for icons

Extension icons keep their own `icons/` convention (the `16/32/48/128.png` size map that feeds `manifest.icons`). See [Project structure](/docs/project-structure#icons). `public/` is for everything else.

## Reserved names

A public file is skipped with a warning if its name would overwrite a generated output: `manifest.json`, a surface's `.html` or `.js`, or anything under `icons/`. Rename it to ship it. Generated output always wins.

## What ships

```
dist/
├── manifest.json
├── popup.html
├── popup.js
├── logo.svg            # from public/
├── img/
│   └── banner.png      # from public/img/
└── ...
```

The same files land in the dev build, so `getURL` and root-relative references resolve identically in `extro dev` and `extro build`.
